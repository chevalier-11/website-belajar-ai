Pemrograman Berorientasi Objek.

Sejauh ini kamu menulis kode sebagai deretan instruksi dan fungsi terpisah. Pemrograman berorientasi objek *(object-oriented programming)* menawarkan cara berpikir yang berbeda: kumpulkan data dan fungsi yang berkaitan ke dalam satu unit bernama kelas. Hampir semua library kecerdasan buatan — dari scikit-learn sampai PyTorch — dibangun di atas paradigma ini, jadi memahaminya bukan pilihan, melainkan keharusan.

8.1 Konsep Kelas dan Objek

Mari mulai dari analogi sederhana. Bayangkan sebuah cetak biru *(blueprint)* rumah — dokumen teknis yang mendeskripsikan semua detail: jumlah kamar, posisi pintu, ukuran jendela. Cetak biru itu bukan rumah yang sebenarnya, ia hanya deskripsi. Tapi dari satu cetak biru, kamu bisa membangun banyak rumah nyata — setiap rumah punya alamat berbeda, penghuninya berbeda, kondisinya berbeda, tapi semua mengikuti desain yang sama.

Di Python, kelas *(class)* adalah cetak biru itu. Objek *(object)* adalah rumah nyata yang dibangun dari cetak biru tersebut. Setiap objek yang dibuat dari kelas yang sama berbagi struktur dan kemampuan yang sama, tapi punya data sendiri yang independen. Proses membuat objek dari kelas disebut instansiasi *(instantiation)*, dan objek yang dihasilkan disebut instans *(instance)*.

python 
# Mendefinisikan kelas — ini adalah cetak biru, belum jadi objek
class ModelML:
    """Kelas sederhana untuk merepresentasikan model ML."""
    pass   # 'pass' sebagai penampung sementara

# Membuat objek (instansiasi) dari kelas
model_a = ModelML()   # objek pertama
model_b = ModelML()   # objek kedua — terpisah dari model_a

print(type(model_a))   # → <class '__main__.ModelML'>
print(model_a is model_b)   # → False — dua objek berbeda di memori

# Kamu bisa menambahkan atribut ke objek secara langsung (tidak dianjurkan)
model_a.nama     = "XGBoost"
model_a.akurasi  = 0.93
model_b.nama     = "Random Forest"
model_b.akurasi  = 0.91

print(model_a.nama)    # → XGBoost
print(model_b.nama)    # → Random Forest
# model_a dan model_b punya data yang sepenuhnya terpisah

Kamu mungkin bertanya: kalau bisa tambah atribut langsung ke objek seperti di atas, kenapa butuh kelas? Jawabannya: kelas memaksa konsistensi. Tanpa kelas yang mendefinisikan struktur, setiap objek bisa punya atribut yang berbeda-beda — satu punya nama, yang lain punya name, yang ketiga tidak punya sama sekali. Kelas memastikan semua objek yang dibuat darinya dimulai dengan struktur yang sama dan terdefinisi dengan baik.

💡
Semua Benda di Python adalah Objek

Ini bukan kiasan — setiap nilai di Python, termasuk integer, string, list, fungsi, dan bahkan kelas itu sendiri, adalah objek. Itulah kenapa "halo".upper() bisa dipanggil: string adalah objek dari kelas str, dan upper adalah metode yang didefinisikan di kelas tersebut. Saat kamu menulis [1,2,3].append(4), kamu sedang memanggil metode append dari objek bertipe list.

8.2 Atribut dan Metode

Kelas menggabungkan dua hal: atribut (data yang dimiliki objek) dan metode (fungsi yang bisa dilakukan objek). Metode pada dasarnya adalah fungsi biasa yang didefinisikan di dalam kelas, dengan satu perbedaan krusial: parameter pertamanya selalu self — referensi ke objek itu sendiri. Lewat self inilah sebuah metode bisa mengakses dan mengubah atribut objeknya sendiri.

Konvensi nama self sebenarnya bukan kata kunci Python yang diwajibkan — secara teknis kamu bisa menamai parameter pertama itu apapun. Tapi komunitas Python sudah sepakat untuk selalu menggunakannya, jadi ikuti konvensi ini tanpa terkecuali agar kode mudah dibaca oleh siapapun.

python 
class DatasetML:
    """Merepresentasikan sebuah dataset untuk keperluan ML."""

    # Atribut kelas — dibagi semua instans dari kelas ini
    versi_format = "1.0"

    # Metode — fungsi di dalam kelas
    # 'self' wajib jadi parameter pertama, merujuk ke objek itu sendiri
    def muat_data(self, data):
        """Muat data ke dalam dataset."""
        self.data = data          # atribut instans — hanya milik objek ini
        self.jumlah_baris = len(data)
        self.jumlah_kolom = len(data[0]) if data else 0

    def info(self):
        """Tampilkan ringkasan dataset."""
        print(f"Dataset: {self.jumlah_baris} baris × {self.jumlah_kolom} kolom")
        print(f"Format versi: {self.versi_format}")   # akses atribut kelas

    def kepala(self, n=5):
        """Tampilkan n baris pertama."""
        for baris in self.data[:n]:
            print(baris)

# Membuat objek dan memanggil metodenya
ds1 = DatasetML()
ds1.muat_data([
    [1.2, 3.4, 0],
    [2.1, 1.5, 1],
    [0.8, 4.2, 0],
])
ds1.info()    # → Dataset: 3 baris × 3 kolom
ds1.kepala(2)  # → dua baris pertama

# ds2 terpisah sepenuhnya dari ds1
ds2 = DatasetML()
ds2.muat_data([[5, 6, 7, 1], [8, 9, 0, 0]])
ds2.info()    # → Dataset: 2 baris × 4 kolom

Atribut Kelas vs Atribut Instans

Ada dua jenis atribut yang perlu kamu bedakan. Atribut kelas didefinisikan langsung di badan kelas (di luar metode apapun) dan nilainya dibagi oleh semua objek yang dibuat dari kelas itu. Atribut instans dibuat di dalam metode menggunakan self.nama_atribut dan nilainya unik untuk setiap objek. Kalau kamu ubah atribut kelas lewat instans, Python sebenarnya membuat atribut instans baru yang "menutupi" atribut kelas untuk objek itu saja:

python 
class Konfigurasi:
    # Atribut kelas — nilai default yang dibagi semua instans
    learning_rate = 0.001
    batch_size    = 32
    epochs        = 100

config_a = Konfigurasi()
config_b = Konfigurasi()

# Keduanya awalnya berbagi nilai yang sama
print(config_a.learning_rate)   # → 0.001
print(config_b.learning_rate)   # → 0.001

# Ubah atribut KELAS lewat nama kelas — mempengaruhi semua instans
Konfigurasi.epochs = 200
print(config_a.epochs)   # → 200
print(config_b.epochs)   # → 200

# Ubah lewat instans — hanya membuat atribut instans baru untuk config_a
config_a.learning_rate = 0.01
print(config_a.learning_rate)   # → 0.01  (atribut instans menutupi atribut kelas)
print(config_b.learning_rate)   # → 0.001 (tidak terpengaruh)
⚠️
Atribut Kelas yang Mutable Bisa Jadi Jebakan

Sama seperti jebakan mutable default di fungsi (Bab 6), atribut kelas bertipe list atau dictionary dibagi semua instans. Kalau satu instans mengubah isinya (bukan menggantinya), semua instans terpengaruh. Hindari mendefinisikan list atau dictionary sebagai atribut kelas. Buat mereka sebagai atribut instans di dalam __init__.

8.3 Konstruktor __init__

Di contoh sebelumnya, atribut instans baru tersedia setelah memanggil muat_data() — artinya objek yang baru saja dibuat belum punya atribut apapun. Ini masalah: kamu bisa saja lupa memanggil muat_data() dan objeknya berada dalam kondisi yang tidak terdefinisi dengan baik. Solusinya adalah metode khusus __init__, yang secara otomatis dipanggil Python setiap kali objek baru dibuat.

Metode __init__ sering disebut konstruktor — tugasnya menginisialisasi objek baru dengan nilai awal yang bermakna. Semua atribut instans seharusnya didefinisikan di sini, bukan tersebar di berbagai metode. Ini membuat struktur kelas langsung terlihat jelas hanya dengan membaca __init__.

python 
class ModelKlasifikasi:
    """Model klasifikasi dengan konfigurasi yang lengkap sejak dibuat."""

    def __init__(self, nama, learning_rate=0.001, max_iter=100):
        """
        Inisialisasi model baru.

        Parameter:
            nama          (str)   : nama model
            learning_rate (float) : laju belajar, default 0.001
            max_iter      (int)   : maksimum iterasi, default 100
        """
        # Atribut dari parameter — dikustomisasi saat membuat objek
        self.nama          = nama
        self.learning_rate = learning_rate
        self.max_iter      = max_iter

        # Atribut dengan nilai awal tetap — bukan dari parameter
        self.sudah_dilatih = False
        self.akurasi       = None
        self.riwayat_loss  = []     # list baru untuk setiap instans — aman!

    def latih(self, X, y):
        """Simulasi pelatihan model."""
        print(f"Melatih {self.nama} dengan lr={self.learning_rate}...")
        for i in range(1, self.max_iter + 1):
            loss = 1.0 / i   # simulasi loss yang menurun
            self.riwayat_loss.append(round(loss, 4))
        self.sudah_dilatih = True
        self.akurasi       = 0.90 + self.learning_rate   # simulasi
        print(f"Selesai. Akurasi: {self.akurasi:.1%}")

    def prediksi(self, X):
        """Lakukan prediksi (hanya bisa setelah dilatih)."""
        if not self.sudah_dilatih:
            raise RuntimeError(f"Model '{self.nama}' belum dilatih. Panggil latih() dulu.")
        print(f"Memprediksi {len(X)} sampel dengan {self.nama}...")
        return [1 if x[0] > 0.5 else 0 for x in X]   # simulasi sederhana

    def ringkasan(self):
        """Tampilkan ringkasan model."""
        status = "Sudah dilatih" if self.sudah_dilatih else "Belum dilatih"
        print(f"=== {self.nama} ===")
        print(f"  LR       : {self.learning_rate}")
        print(f"  Max iter : {self.max_iter}")
        print(f"  Status   : {status}")
        if self.akurasi:
            print(f"  Akurasi  : {self.akurasi:.1%}")

# Buat dua model berbeda dari kelas yang sama
m1 = ModelKlasifikasi("Regresi Logistik")
m2 = ModelKlasifikasi("SVM", learning_rate=0.01, max_iter=200)

m1.ringkasan()   # belum dilatih
m1.latih([[0.3], [0.7]], [0, 1])
m1.ringkasan()   # sudah dilatih

# m2 tidak terpengaruh
print(m2.sudah_dilatih)    # → False
print(m2.riwayat_loss)     # → [] — list kosong milik m2 sendiri

Metode Dunder Lainnya yang Berguna

Python punya banyak metode khusus yang namanya diawali dan diakhiri dua garis bawah — sering disebut metode dunder *(double underscore)* atau "magic method". Kamu sudah kenal __init__. Dua lainnya yang sangat berguna untuk membuat objek mudah digunakan adalah __str__ (menentukan tampilan teks objek saat dicetak) dan __repr__ (tampilan representasi teknis untuk debugging):

python 
class Vektor:
    """Vektor dua dimensi sederhana."""

    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __str__(self):
        """Tampilan ramah pengguna — dipanggil oleh print()."""
        return f"Vektor({self.x}, {self.y})"

    def __repr__(self):
        """Tampilan teknis — berguna untuk debugging di konsol."""
        return f"Vektor(x={self.x!r}, y={self.y!r})"

    def __add__(self, lain):
        """Memungkinkan operasi + antar dua Vektor."""
        return Vektor(self.x + lain.x, self.y + lain.y)

    def panjang(self):
        """Hitung panjang (magnitudo) vektor."""
        return (self.x ** 2 + self.y ** 2) ** 0.5

v1 = Vektor(3, 4)
v2 = Vektor(1, 2)

print(v1)          # → Vektor(3, 4)    ← __str__ dipanggil
print(repr(v1))    # → Vektor(x=3, y=4) ← __repr__ dipanggil
v3 = v1 + v2       # → __add__ dipanggil
print(v3)          # → Vektor(4, 6)
print(v1.panjang()) # → 5.0

8.4 Pewarisan (Inheritance) Dasar

Pewarisan adalah mekanisme yang memungkinkan satu kelas "mewarisi" semua atribut dan metode dari kelas lain, lalu menambahkan atau mengubah bagian-bagian tertentu. Kelas yang diwarisi disebut kelas induk *(parent class)* atau kelas dasar, sementara kelas yang mewarisi disebut kelas anak *(child class)* atau subkelas.

Ini mencerminkan hubungan "adalah-sebuah" *(is-a)*: seekor Kucing adalah sebuah Hewan. Sebuah RandomForest adalah sebuah ModelEnsemble, yang merupakan sebuah ModelML. Pewarisan menghindari duplikasi kode — semua logika yang berlaku untuk kategori umum ditulis sekali di kelas induk, dan kelas anak hanya perlu menambahkan atau menyesuaikan perbedaannya:

python 
# Kelas induk — mendefinisikan perilaku umum semua model
class ModelDasar:
    """Kelas dasar untuk semua model ML."""

    def __init__(self, nama):
        self.nama          = nama
        self.sudah_dilatih = False

    def periksa_sudah_dilatih(self):
        """Lempar error jika model belum dilatih."""
        if not self.sudah_dilatih:
            raise RuntimeError(f"'{self.nama}' belum dilatih.")

    def simpan(self, path):
        """Simulasi menyimpan model ke file."""
        self.periksa_sudah_dilatih()
        print(f"Model '{self.nama}' disimpan ke {path}")

    def __str__(self):
        status = "terlatih" if self.sudah_dilatih else "belum dilatih"
        return f"<{self.__class__.__name__}: {self.nama} ({status})>"


# Kelas anak — mewarisi ModelDasar, tambah fitur spesifik regresi
class ModelRegresi(ModelDasar):   # nama kelas induk di dalam kurung
    """Model untuk tugas regresi."""

    def __init__(self, nama, learning_rate=0.01):
        super().__init__(nama)   # panggil __init__ kelas induk
        self.learning_rate = learning_rate
        self.koefisien     = None

    def latih(self, X, y):
        """Latih model regresi."""
        print(f"Melatih regresi '{self.nama}'...")
        self.koefisien     = [0.5] * len(X[0])   # simulasi koefisien
        self.sudah_dilatih = True
        print("Selesai.")

    def prediksi(self, X):
        """Prediksi nilai kontinu."""
        self.periksa_sudah_dilatih()   # metode warisan dari ModelDasar
        return [sum(xi) * 0.5 for xi in X]   # simulasi prediksi


# Kelas anak lain — model klasifikasi
class ModelKlasifikasi(ModelDasar):
    """Model untuk tugas klasifikasi."""

    def __init__(self, nama, n_kelas=2):
        super().__init__(nama)
        self.n_kelas = n_kelas

    def latih(self, X, y):
        print(f"Melatih klasifikasi '{self.nama}' ({self.n_kelas} kelas)...")
        self.sudah_dilatih = True
        print("Selesai.")

    def prediksi(self, X):
        self.periksa_sudah_dilatih()
        return [0 if sum(xi) < 1 else 1 for xi in X]

# Semua model punya metode 'simpan' dari kelas induk
reg = ModelRegresi("LinearReg", learning_rate=0.05)
clf = ModelKlasifikasi("LogisticReg")

reg.latih([[1, 2], [3, 4]], [2.1, 3.9])
reg.simpan("model_reg.pkl")    # metode dari kelas induk!
print(reg)                     # → <ModelRegresi: LinearReg (terlatih)>

# Periksa hubungan pewarisan
print(isinstance(reg, ModelRegresi))   # → True
print(isinstance(reg, ModelDasar))     # → True — reg JUGA adalah ModelDasar!
print(isinstance(reg, ModelKlasifikasi)) # → False
💡
super() — Memanggil Metode Kelas Induk

Fungsi super() memberikan akses ke kelas induk. Paling sering dipakai di __init__ kelas anak untuk memanggil __init__ kelas induk sebelum menambahkan atribut baru. Tanpa ini, atribut yang didefinisikan di __init__ kelas induk tidak akan ada di objek hasil kelas anak. Selalu panggil super().__init__() di baris pertama __init__ kelas anak.

Penimpaan Metode (Method Overriding)

Kelas anak bisa "menimpa" *(override)* metode yang sudah ada di kelas induk dengan mendefinisikan ulang metode dengan nama yang sama. Saat metode itu dipanggil di objek kelas anak, versi kelas anak yang dijalankan, bukan versi kelas induk. Ini yang terjadi di contoh sebelumnya — baik ModelRegresi maupun ModelKlasifikasi mendefinisikan metode latih() mereka sendiri yang berbeda:

python — polimorfisme 
# Polimorfisme: satu nama, banyak perilaku
# Semua model punya .latih() dan .prediksi(), tapi cara kerjanya beda

model_list = [
    ModelRegresi("Ridge"),
    ModelKlasifikasi("SVM"),
    ModelKlasifikasi("KNN", n_kelas=3),
]

X_latih = [[1, 2], [3, 4], [0, 1]]
y_latih = [0, 1, 0]

# Kode yang sama bekerja untuk semua jenis model
# karena semua mewarisi dari ModelDasar dan punya antarmuka yang sama
for model in model_list:
    model.latih(X_latih, y_latih)   # tiap model punya implementasi sendiri
    model.simpan(f"{model.nama}.pkl")   # metode warisan, sama untuk semua
    print(model)                          # __str__ dari ModelDasar
    print()

8.5 Kapan OOP Membantu dan Kapan Tidak Perlu

OOP adalah alat, bukan tujuan. Salah satu kesalahan yang sering dibuat pemula yang baru belajar OOP adalah mencoba membungkus segalanya dalam kelas, bahkan ketika fungsi biasa sudah lebih dari cukup. Kode yang dipaksakan jadi "OOP" tanpa alasan yang jelas justru menjadi lebih sulit dibaca dan dipahami.

Pertanyaan yang perlu kamu ajukan sebelum membuat kelas: "Apakah ada data yang perlu dibawa bersama, dan operasi yang berkaitan erat dengan data itu, yang perlu diwakili sebagai satu kesatuan yang punya identitas?" Jika jawabannya ya, kelas mungkin tepat. Jika kamu hanya butuh mengeksekusi satu tugas atau transformasi, fungsi biasa hampir selalu lebih sederhana dan lebih bersih.

Gunakan Kelas ketika... Gunakan Fungsi ketika...
Ada beberapa item data yang terkait erat dan perlu dibawa bersama Kamu hanya perlu mentransformasi input menjadi output
Objek perlu punya status *(state)* yang berubah sepanjang waktu Tidak ada state yang perlu dijaga antar pemanggilan
Kamu akan membuat banyak instans dengan data yang berbeda-beda Hanya satu operasi yang perlu dilakukan, satu kali
Ada hubungan hierarki yang masuk akal untuk dimodelkan dengan pewarisan Operasinya simpel dan tidak punya variasi yang berarti
Kamu membangun library atau API yang akan dipakai orang lain Kamu sedang menulis skrip analisis sekali pakai

Contoh: Kapan Fungsi Lebih Baik dari Kelas

python — fungsi vs kelas yang berlebihan 
# ── BERLEBIHAN: kelas untuk tugas yang sederhana ───────────────
class PenghitungRata:
    def __init__(self, data):
        self.data = data
    def hitung(self):
        return sum(self.data) / len(self.data)

p = PenghitungRata([85, 90, 78])
print(p.hitung())   # terlalu banyak kode untuk satu operasi sederhana

# ── LEBIH BAIK: fungsi biasa ────────────────────────────────────
def rata_rata(data):
    return sum(data) / len(data)

print(rata_rata([85, 90, 78]))   # jelas, langsung, mudah dipahami

# ── KELAS MASUK AKAL: ketika ada state dan identitas ─────────────
class PencatatEksperimen:
    """Catat dan kelola hasil banyak eksperimen ML."""

    def __init__(self, nama_proyek):
        self.nama_proyek = nama_proyek
        self.eksperimen  = []   # state yang tumbuh seiring waktu

    def tambah(self, nama, params, skor):
        """Catat satu eksperimen baru."""
        self.eksperimen.append({
            "nama": nama, "params": params, "skor": skor
        })

    def terbaik(self):
        """Kembalikan eksperimen dengan skor tertinggi."""
        if not self.eksperimen:
            return None
        return max(self.eksperimen, key=lambda e: e["skor"])

    def laporan(self):
        """Tampilkan semua eksperimen terurut dari yang terbaik."""
        print(f"=== {self.nama_proyek} — {len(self.eksperimen)} eksperimen ===")
        urut = sorted(self.eksperimen, key=lambda e: e["skor"], reverse=True)
        for i, e in enumerate(urut, 1):
            print(f"  #{i} {e['nama']:20} skor={e['skor']:.4f}  params={e['params']}")

# Kelas ini masuk akal karena punya state (eksperimen yang terus bertambah)
# dan identitas (ini adalah kumpulan eksperimen proyek tertentu)
pencatat = PencatatEksperimen("Prediksi Harga Rumah")
pencatat.tambah("Ridge (lr=0.001)",  {"lr": 0.001}, 0.8812)
pencatat.tambah("Ridge (lr=0.01)",   {"lr": 0.01},  0.9134)
pencatat.tambah("XGBoost (default)", {"n": 100},    0.9441)
pencatat.laporan()
print("\nEksperimen terbaik:", pencatat.terbaik())

OOP di Library ML: Apa yang Sudah Kamu Gunakan Tanpa Sadar

Mungkin kamu tidak menyadarinya, tapi kamu sudah berinteraksi dengan OOP sejak awal belajar Python. Setiap kali kamu memanggil "halo".upper(), kamu memanggil metode upper dari objek bertipe str. Setiap kali kamu memanggil [1,2,3].append(4), kamu memanggil metode dari objek bertipe list. Nanti di Buku 3, kamu akan menggunakan scikit-learn seperti ini:

python — preview: OOP di scikit-learn 
# Ini adalah kode scikit-learn yang akan kamu tulis di Buku 3
# Sekarang cukup baca dan pahami polanya

# from sklearn.linear_model import LogisticRegression
#
# model = LogisticRegression(max_iter=200, C=1.0)  ← instansiasi dengan __init__
# model.fit(X_train, y_train)                       ← memanggil metode .fit()
# prediksi = model.predict(X_test)                  ← memanggil metode .predict()
# akurasi  = model.score(X_test, y_test)            ← memanggil metode .score()
# koef     = model.coef_                            ← mengakses atribut .coef_

# Pola yang SAMA untuk semua model di scikit-learn:
